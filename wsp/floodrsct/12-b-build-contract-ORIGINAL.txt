Yes — the audit classification is right, but I would not “create everything in parallel” until the data contracts are frozen. Otherwise the fetch scripts and build_event_dataset.py will drift.

I would structure the work like this:

1. Lock the feature contract first

Create one table like this before coding:

feature_name
scenario_id
source_type: public_fetch | derived | operational_unknown
source_dataset
raw_s3_path
processed_s3_path
build_function
output_column
missing_behavior
field_status_behavior
version_target

This becomes the checklist for v24.003.

2. Public fetch scripts: separate raw fetch from derived build

For fetchable public data, use small independent scripts:

fetch_nlcd_impervious.py
fetch_dem_elevation.py
fetch_mtbs_burn_scars.py
fetch_nhdplus_catchments.py
fetch_mta_stations.py

Each script should write:

s3://.../raw/<dataset>/<version>/
s3://.../manifests/<dataset>/<version>/manifest.json

The manifest should include:

{
  "dataset": "nlcd_impervious",
  "version": "v24.003",
  "source_url": "...",
  "fetched_at": "...",
  "s3_key": "...",
  "etag": "...",
  "sha256": "...",
  "crs": "...",
  "license": "...",
  "notes": "..."
}

Do not bake ZCTA aggregation into the fetch scripts. Fetch scripts should only acquire and normalize raw inputs.

3. Derived features belong in build_event_dataset.py

These should be build steps, not fetch steps:

SLOSH → ZCTA aggregation
sewershed → ZCTA join
coastal distance
levee → ZCTA join
catchment/wash → ZCTA join
burn scar → ZCTA overlay
impervious surface → ZCTA aggregation
DEM elevation → ZCTA aggregation

Use one function per feature family:

build_impervious_features()
build_elevation_features()
build_burn_scar_features()
build_catchment_features()
build_slush_or_slosh_features()
build_sewershed_features()
build_levee_features()
build_coastal_distance_features()

Each function should be idempotent and return a dataframe keyed by:

scenario_id
zcta_id
dataset_version
4. Operational/real-time fields should be explicitly unknown

For these:

drainage_capacity_status
road_access_status
evacuation_route_status
pump_station_status if not available historically

Do not pretend they are missing public data. They are operational status fields.

Use:

{
  "value": "unknown",
  "field_status": {
    "status": "operational_status_unavailable",
    "missing_reason": "Historical public archive does not provide real-time operational status for this field.",
    "expected_available_version": null
  }
}

For F40, if unknown prevents an ordinal mapping, then the vector field should become:

{
  "access_fragility": null,
  "field_status": {
    "access_fragility": {
      "status": "operational_status_unavailable",
      "missing_reason": "Required operational route/drainage status is unavailable for historical replay.",
      "expected_available_version": null
    }
  }
}

This is better than saying missing_source_data for everything. It tells the operator the truth: some values are not fetchable from public history.

5. Scenario-specific implications
S1 Houston:
  Fetch NLCD impervious.
  Derive bayou/drainage overlays if available.
  drainage_capacity_status = unknown unless you have operational archive.

S2 New Orleans:
  Derive levee → ZCTA and canal/pump proximity.
  pump_station_status likely unknown unless an operational archive exists.

S3 Riverside/Coachella:
  Fetch MTBS burn scars, DEM, NHDPlus/catchments.
  road_access_status = unknown for historical replay.

S4 Lee/Charlotte:
  Derive SLOSH → ZCTA, coastal distance, elevation.
  evacuation_route_status = unknown for historical replay.

S5 NYC/NJ:
  Fetch MTA stations.
  Derive sewershed → ZCTA if source exists.
  access_fragility remains null until sewer-shed load + subway mapping are complete.
6. Implementation order

I would do it in this order:

1. Add feature contract YAML/JSON.
2. Add raw fetch scripts.
3. Add S3 manifest writer.
4. Add derived builders to build_event_dataset.py.
5. Add field_status generation.
6. Add F40 source_map validation.
7. Add tests for each scenario’s required columns.

The key rule:

Fetch in parallel, but derive through one controlled build contract.

That prevents the dataset from becoming a pile of unrelated columns that F40 cannot safely explain.
